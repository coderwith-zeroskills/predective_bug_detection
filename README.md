# Bug Risk Predictor

A machine learning system that reads Jira story text before sprint commitment and flags high-risk stories likely to produce bugs. Built with XGBoost and NLP feature extraction.

---

## The Problem

Engineering teams commit to sprint stories without knowing which ones carry high bug risk. A story like *"Migrate user authentication to OAuth2"* touches core infrastructure shared by multiple services — but it looks identical in format to *"Update button color on landing page"*. Manual review is slow, inconsistent, and skipped under deadline pressure.

## The Solution

This model reads story title, description, story points, and comment count — and outputs a risk score between 0% and 100% in under a second.

```
====================================================
  JIRA STORY RISK ASSESSMENT
====================================================
  Title        : Migrate user authentication to OAuth2
  Story points : 13  |  Comments: 12
----------------------------------------------------
  Risk score   : 98.3%
  [###################-]
  [!!!] HIGH RISK — flag before sprint commitment
  Risk signals  : auth, migrat, oauth
====================================================
```

## How It Works

```
Jira story text
      │
      ▼
Text cleaning          remove stopwords, lowercase, strip punctuation
      │
      ├──▶ TF-IDF (500 features)      word importance scores across corpus
      ├──▶ Numeric features (2)        story points + comment count, scaled
      └──▶ Keyword flags (15)          domain signals: auth, payment, schema...
                    │
                    ▼
           Feature matrix (517 features)
                    │
                    ▼
            XGBoost Classifier
            scale_pos_weight=2     biased toward catching all risky stories
            threshold=0.35         flag if 35%+ probability of risk
                    │
                    ▼
            Risk score + verdict
```

**Why XGBoost?**
It handles mixed feature types (sparse TF-IDF + dense numeric) natively, is fast to train on CPU, and is the most commonly used model for tabular data in industry.

**Why optimize for Recall over Precision?**
Missing a risky story (false negative) costs the team days of debugging post-sprint. A false alarm (flagging a safe story) costs 5 minutes of review. The business cost is asymmetric, so recall is the right metric to maximize.

**Why lower the threshold to 0.35?**
XGBoost's default decision threshold is 0.5. Lowering it to 0.35 means the model flags a story as risky if there is even a 35% probability of risk. This trades some precision (more false alarms) for higher recall (fewer missed risky stories) — the correct tradeoff for this use case.

## Quickstart

**1. Clone and set up environment**

```bash
git clone https://github.com/your-username/bug-predictor.git
cd bug-predictor

python -m venv venv

# Windows
venv\Scripts\activate

# Mac / Linux
source venv/bin/activate

pip install -r requirements.txt
```

**2. Run the full pipeline**

```bash
python generate_data.py       # Step 1: create dataset
python feature_extraction.py  # Step 2: extract features
python train_model.py         # Step 3: train and evaluate model
python predict.py             # Step 4: run predictions
```

**3. Predict on your own story**

Open `predict.py` and call `predict_risk()` with your own values:

```python
predict_risk(
    title="Your Jira story title here",
    description="Full story description and acceptance criteria",
    story_points=8,
    num_comments=6
)
```

## Key Concepts Demonstrated

- **NLP feature extraction** — converting raw text into numeric features using TF-IDF with unigrams and bigrams
- **Domain knowledge features** — hand-crafted keyword flags based on engineering intuition, combined with learned features
- **Imbalanced classification** — using `scale_pos_weight` and threshold tuning to optimize for recall over precision
- **Model persistence** — saving and reloading vectorizer, scaler, and model for production use
- **Cross-validation** — 5-fold stratified CV to confirm results are not a product of a lucky train/test split
